# 產品需求文件 (PRD) - PgM 週報系統

## 1. 專案概述
- **專案名稱**：VIA Technologies PgM Weekly Report System (PgM 週報系統)
- **文件版本**：v1.0
- **撰寫日期**：2026/04/06
- **產品簡介**：本系統為一輕量級、易於維護的內部週報展示平台。旨在將以 Markdown 格式撰寫的 PgM 週報，轉換為易於閱讀的網頁介面，方便跨部門協作與高層管理團隊快速掌握專案進度與瓶頸。

## 2. 目標與願景
- **痛點解決**：過去週報可能散落在不同平台或以純文字/Markdown檔案流通，缺乏統一且易於回溯歷史版本的友善介面。
- **產品目標**：
  1. 提供單一入口掌握所有歷史與最新週報。
  2. 達成「零資料庫、低維護成本」的架構，僅透過上傳 Markdown 檔案即可發布新週報。
  3. 支援自動化部署，減輕維運負擔。

## 3. 目標使用者
1. **專案經理 (PgM) / 產品經理 (PM)**：產出與上傳週報內容的建立者。
2. **高層主管 / 利益關係人 (Stakeholders)**：需要快速瀏覽最新進度、重要里程碑與專案風險的閱讀者。
3. **專案成員**：需同步專案資訊與待辦事項的其他部門成員。

## 4. 核心功能需求

### 4.1. 前台閱覽介面
- **側邊欄導覽 (Sidebar)**：
  - 應能列出目前系統內所有的週報版本（例如：v1, v2, v8）。
  - 自動解析檔案內容，顯示對應的「報告週期」及「報告日期」。
  - 預設載入並標示最新版本的週報。
- **內容展示區 (Main Content)**：
  - 將 Markdown 文本即時渲染成豐富的 HTML 格式。
  - 支援標題、清單、粗體/斜體、表格及程式碼區塊等標準 Markdown 語法。
  - 提供良好的閱讀體驗（排版、間距、動線）。

### 4.2. 資料更新介面 (發布流程)
- **檔案驅動機制 (File-driven)**：
  - 系統不設置複雜的後台編輯器與關聯式資料庫。
  - PgM 將新寫好的 `_vN.md` 檔案放入指定目錄 (`backend/reports/`) 即代表完成發布準備。
- **自動化解析**：
  - 系統自動解析檔名（如 `MediaAgent_週報_v8.md` 中提取 `v8`）。
  - 系統要求 Markdown 檔案內部必須包含特定格式（如 `報告週期：YYYY/MM/DD – YYYY/MM/DD`），後端自動抓取此 meta 資訊以供前端選單顯示。

### 4.3. 部署與效能
- **自動化部署 (CI/CD)**：
  - 整合 GitHub 搭配 Railway，實現 Push 即 Deploy。PgM 將檔案推送到 main branch 後，約 2 分鐘內自動生效。
- **單一服務架構 (Single Service)**：
  - Express.js 同時作為 API 伺服器與 React 靜態檔案伺服器，節省維運與雲端資源費用。

## 5. 系統架構設計與技術指標
- **前端架構**：React 18 + Vite，使用 `marked` 等套件進行 Markdown 渲染。
- **後端架構**：Node.js + Express，提供對前端的 RESTful API 服務（如：`/api/reports` 取回週報清單與內容）。
- **基礎設施**：Railway 平台 (Docker container化部署)。

## 6. 上線與驗收標準
1. **渲染正確性**：各種 Markdown 語法包含自訂表格與圖表，均可在瀏覽器內正常無錯位呈現。
2. **自動化順暢度**：將新的 `.md` 檔案推上 GitHub 後，網頁能夠在 3 分鐘內透過 Railway 更新。
3. **無效資料處理**：當 Markdown 檔名格式不正確或缺少特定 tag 時，系統應有容錯機制（例如不顯示或顯示預設值），且不會導致系統崩潰。
4. **效能指標**：網頁初始載入時間於正常網路環境下需小於 2 秒。

## 7. 未來與後續迭代 (Phase 2 & beyond)
- **全文搜尋功能**：能夠根據關鍵字檢索過去所有週報內的項目。
- **自動產生圖表**：將週報內的進度數據，自動視覺化為進度條或燃盡圖。
- **權限與安全性**：增加登入機制 (SSO) 以限制只有內部員工可存取。
- **評論與標記**：允許使用者在特定段落留言或是 @ 其他成員。
